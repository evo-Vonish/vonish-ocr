# 模型缓存清理

长期使用后，模型文件与缓存可能占用大量磁盘空间。VonishOCR 提供多种清理方式。

## 显存释放

模型加载后驻留显存，即使不识别时也可能保持占用。显存释放方式：

| 方式 | 操作路径 | 效果 |
|------|---------|------|
| 切换至节能模式 | 状态条 → 节能模式 | 降低并发，减少显存峰值 |
| 卸载当前模型 | 后端控制台 → 模型管理 → 卸载 | 释放当前模型占用的显存 |
| 清理全部缓存 | 后端控制台 → 清理缓存 | 卸载所有模型，释放全部显存 |
| 退出应用 | 关闭窗口或退出托盘 | 进程结束，显存自然释放 |

> 注意：DirectML 使用 WDDM 驱动，显存分配由操作系统管理。`gc.collect()` 只释放 Python 对象，不保证 GPU 内存立即回收。ONNX Runtime 的 arena 分配器会保留已分配的 GPU 内存供下次复用。

## 磁盘清理

### 模型文件

模型文件位于 `models/` 目录，占用空间：

| 模型 | 大小 |
|------|------|
| rapidocr-mobile-cn | 22 MB |
| cnocr-standard-cn | 80 MB |
| paddleocr-vl-1.5 | 1.87 GB |

若不再使用某档模型，可直接删除对应 `.onnx` 文件。`manifest.json` 中的记录可保留，应用会自动跳过未找到文件的模型。

### ONNX 缓存

ONNXRuntime 在首次运行时会生成优化后的模型缓存（如常量折叠、算子融合），位于：

- Windows：`%TEMP%/onnxruntime/`
- macOS：`/tmp/onnxruntime/`
- Linux：`/tmp/onnxruntime/`

这些缓存可安全删除，下次加载模型时会自动重建。

### 日志归档

日志文件位于 `logs/` 目录，按 10MB 大小轮转，默认保留 5 个文件。长期积累的日志可能达到数百 MB。

清理方式：

1. 后端控制台 → 日志审计 → 「归档并清理」
2. 系统会将超过 30 天的日志压缩为 `.gz` 并移至 `logs/archive/`
3. 超过 90 天的归档日志自动删除

## 后端控制台「清理缓存」按钮

后端控制台提供一键清理功能，执行以下操作：

1. 卸载所有驻留模型
2. 触发 Python `gc.collect()`
3. 清除 ONNXRuntime 临时缓存
4. 压缩超过 30 天的日志
5. 显示清理前后的磁盘空间对比

清理操作需要二次确认（危险操作，输入「清理」解锁）。

---

> 缓存清理的设计原则是「可控的释放」。用户决定何时清理、清理什么，系统只提供工具与信息，不做自动删除。显存占用提示与磁盘空间统计帮助用户做出知情决策。
