# 诊断与排错

当 VonishOCR 出现行为异常时，按照以下流程进行系统诊断。

## 常见问题 FAQ

### 模型加载失败

**现象**：启动后顶部状态条显示「后端异常」，左侧模型区域显示「模型未加载」。

**排查**：
1. 检查 `models/` 目录是否存在 `.onnx` 文件
2. 检查 `manifest.json` 中的 `sha256` 与实际文件是否匹配
3. 查看 `logs/backend.log` 中是否有 `ModelCorruptedError` 或 `ModelFileMissingError`
4. 确认 ONNXRuntime 的动态库是否正确加载（Windows 下检查 `onnxruntime_providers_shared.dll`）

**解决**：重新下载模型文件，或手动校验 SHA256。

### 识别结果空白

**现象**：拖入图片后，ResultPanel 显示提示文字而非识别结果。

**排查**：
1. 确认 `has result?` 状态（见下方诊断代码）
2. 检查走的是单图路径还是批量路径
3. 在 `tauri_ipc.js` 中加 log，确认 IPC 返回格式
4. 用 cURL 直接调用后端，确认返回 JSON 中有 `"text"` 字段

**解决**：若 id 不匹配，可能是 Bug 8（两套 id 系统）；若后端返回空 text，检查场景分类器是否误判为空白图片。

### API 无响应

**现象**：调用 `localhost:8000/v1/ocr` 时超时或返回 502。

**排查**：
1. 确认 Python sidecar 进程存活（任务管理器查找 `python.exe`）
2. 检查 `config.json` 中的端口配置是否与请求一致
3. 查看后端日志是否有 `Uvicorn` 启动错误
4. 确认防火墙未拦截本地回环地址

**解决**：重启应用，或在后端控制台手动重启 API 服务。

### 显存不足

**现象**：批量处理时 GPU 内存暴涨，进度条卡住。

**排查**：
1. 打开后端控制台，查看 GPU 显存占用
2. 检查当前并发数（野兽模式 8 / 均衡 4 / 节能 2）
3. 确认 `gc.collect()` 是否正常执行

**解决**：切换至节能模式，或降低并发数至 1-2。

## 导出诊断包

当需要向开发者报告问题时，可导出诊断包：

1. 点击「帮助 → 导出诊断包」
2. 系统自动收集：
   - `logs/` 目录下所有日志文件
   - `config.json`（脱敏处理，隐藏 API Key）
   - 当前界面截图
3. 打包为 ZIP，保存至桌面
4. 用户可直接将 ZIP 上传至 GitHub Issue

## GitHub Issue 提交指南

提交 Issue 前，请确认：

- [ ] 已搜索现有 Issue，避免重复报告
- [ ] 已使用最新版本复现问题
- [ ] 已尝试重启应用与后端
- [ ] 已导出诊断包并准备上传
- [ ] 已按照 Issue 模板填写必要信息

---

> 诊断与排错文档的核心价值在于「自助」。80% 的常见问题可以通过本文档独立解决，只有涉及底层 bug 时才需要联系开发者。这不仅减少了支持负担，也让用户对自己的工具有更深的理解。
